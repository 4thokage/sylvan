# Factory-Printed Identity vs Physical State of Owned Copies

Sylvan's card model has two layers with strict separation. Printing (`card_printings`) carries factory-printed identity — anything fixed at manufacture: game, set, collector number, language, finish, art variation, frame, border, rarity. Each distinct combination is a separate Printing row with its own price. Have Stack (`user_cards`) carries physical state that can vary per owned copy — quantity, condition, signed, altered, graded, location, notes. The matcher always reads identity from `card_printings` and state from `user_cards`; there is no third home for any attribute.

The factory-vs-aftermarket test resolves ambiguous cases. A card signed after release (`is_signed`) is Have Stack state; a factory-signed promo (e.g., a Pokémon autograph promo) is a different Printing. Altered-art customs are state; shipped alt-art variants are different Printings. The rule is consistent — if the factory did it, it's identity; if it happened to the physical copy afterward, it's state.

Provider adapters are responsible for normalizing upstream data into this shape. Scryfall exposes one card object with both foil and non-foil prices; the adapter splits that into two Printing rows. Scryfall exposes the same card in English, Italian, and German as separate API objects; the adapter maps each to its own Printing row. The domain never sees a "card object with multiple finishes" or a "card object without language."

This decision supersedes the ad-hoc choices seen earlier (the dead `card_printings.is_foil` column, the `user_cards.is_foil` duplication, the `user_cards.language` duplication). A reader seeing Scryfall combined card objects normalized into per-Finish Printing rows, or seeing `user_cards.language` removed in favor of deriving from `card_printings`, needs this principle to make sense of those choices. Reversing the separation would re-introduce the duplicated-attribute contradiction where neither column is authoritative.
